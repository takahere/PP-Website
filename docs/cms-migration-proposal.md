# CMS移行提案書

## 1. エグゼクティブサマリー

### 提案の要点

| 項目 | 内容 |
|------|------|
| **現状** | WordPress 2環境を分散運用、属人化・品質劣化が進行 |
| **提案** | AI-First静的サイトへ移行、Intelligence Layerで差別化 |
| **移行期間** | 2-4週間（主要機能） |
| **移行コスト** | ¥10-40万（一時費用） |
| **年間削減** | ¥100-230万 |
| **投資回収** | 約1-2ヶ月 |

### 結論

WordPressから静的HTML + AI運用への移行により：
- **運用コスト削減**（年間¥100万以上）
- **運用効率向上**（Slack一言でコンテンツ公開）
- **将来拡張性**（PRM製品への転用可能）

を実現する。

## 2. 現状の課題

### 2-1. WordPressの根本的課題

| 課題 | 現状 | 影響 |
|------|------|------|
| **環境分散** | WP 2環境 + 別CMS（リリースノート） | 管理コスト増大 |
| **属人化** | 担当者ごとに運用方法が異なる | ナレッジ分断 |
| **プラグイン依存** | 変更の影響範囲が読めない | 改修リスク |
| **デザイン制約** | テーマ依存で自由度低い | ブランド表現に限界 |
| **セキュリティ** | PHP + プラグインの脆弱性リスク | 継続的な対応必要 |

### 2-2. AI時代の変化

```
【従来のコンテンツ運用】
人間がCMS GUI操作 → 手動で公開 → 手動で分析 → 手動で改善
                    ↓
            全てに人手が必要

【AI時代のコンテンツ運用】
人間が自然言語で指示 → AIが実行 → AIが分析 → AIが改善提案
                    ↓
            人間は承認のみ
```

**市場の変化:**
- HubSpotが「Loop Marketing」を提唱（2025年）
- Google検索の60%がクリックなしで終了
- 従来のファネル型マーケティングが機能不全

**機会:**
- AI-Firstで運用を再定義できる
- 先行者として市場ポジションを確立できる
- 自社課題解決がPRM製品の差別化になる

## 3. 移行方針

### 3-1. 設計原則

> **DBの役割は公開サイトの表示ではなくインテリジェンス機能のデータ基盤**

この原則により：
- 公開サイトは静的HTMLで高速・安全
- DBはAI/分析のためのデータレイヤー
- レビュー・テスト負担を構造的に軽減

### 3-2. アーキテクチャ概要

```
┌─────────────────────────────────────────────────────────────┐
│                    Layer 1: Public Site                      │
│                 静的HTML (Vercel CDN配信)                     │
│           partner-prop.com / lab.partner-prop.com            │
│                                                              │
│  ・ビルド時にHTMLを生成                                        │
│  ・ランタイムでDB接続なし                                      │
│  ・攻撃面が最小限                                             │
└─────────────────────────────────────────────────────────────┘
                              ↑ ビルド時のみ接続
┌─────────────────────────────────────────────────────────────┐
│                   Layer 2: Data Foundation                   │
│                      Supabase (PostgreSQL)                   │
│                                                              │
│  ・コンテンツのマスターデータ                                   │
│  ・分析データの蓄積                                           │
│  ・AI機能のデータソース                                        │
└─────────────────────────────────────────────────────────────┘
          ↓                                      ↓
┌────────────────────────┐      ┌────────────────────────────┐
│  Layer 3a: 入力ツール   │      │  Layer 3b: Intelligence    │
│                        │      │                            │
│  ・Claude Code         │      │  ・SEO自動改善ループ         │
│  ・Slack連携           │      │  ・ボトルネック診断          │
│  ・Admin Panel         │      │  ・KPIダッシュボード         │
└────────────────────────┘      └────────────────────────────┘
```

### 3-3. 代替案との比較

| 選択肢 | メリット | デメリット | 判定 |
|--------|---------|-----------|------|
| **STUDIO移行** | 学習コスト低 | AI統合不可、また移行したくなる | ✗ |
| **HubSpot移行** | Loop Marketing対応 | HubL制約、$1,500+/月 | ✗ |
| **WP継続** | 変更なし | 課題解決せず、コスト増大 | ✗ |
| **静的HTML + AI** | AI統合、低コスト、高速 | 初期構築必要 | ✓ |

**なぜ自社開発か:**
- 既製品は「CMS + AI機能追加」（後付け）
- 提案は「AI-First + 静的HTML」（根本から異なる）
- cursor.com事例：3日で移行、運用コスト激減

## 4. SEO維持戦略

### 4-1. 基本方針

**SEO維持は最優先事項。**

| 項目 | 対応 |
|------|------|
| URL構造 | 完全維持（変更なし） |
| リダイレクト | やむを得ない変更は301で対応 |
| canonical | 全ページで正規URL設定 |
| sitemap | 移行後も正規URLのみ含める |

### 4-2. 移行時の対応

```
Phase 1: 並行運用
┌─────────────┐     ┌─────────────┐
│ WordPress   │     │ Next.js     │
│ (既存)      │     │ (新規)      │
└──────┬──────┘     └──────┬──────┘
       │                   │
       └─────────┬─────────┘
                 ↓
         Vercel Edge
         (パス別ルーティング)

・/lab/* → Next.js
・その他 → WordPress（当面）
```

```
Phase 2: 完全移行
┌─────────────────────────────────────┐
│            Next.js                  │
│  (全ページ静的HTML)                  │
└─────────────────────────────────────┘
         ↓
    Vercel CDN配信

・旧URLは全て301リダイレクト設定
・GSCで404/リダイレクトエラー監視
```

### 4-3. 監視体制

| 指標 | ツール | 頻度 |
|------|--------|------|
| インデックス状況 | GSC | 日次 |
| 順位変動 | GSC + 自社ダッシュボード | 日次 |
| クロールエラー | GSC | 日次 |
| Core Web Vitals | PageSpeed Insights | 週次 |

## 8. 想定される懸念と対応策

### 8-1. 品質評価コストの増大

**懸念:**
> フルスクラッチで作ると品質評価コストが増大する

**対応:**
| 従来のCMS | 提案アーキテクチャ |
|-----------|------------------|
| DB変更→即公開 | DB変更→ビルド実行→公開 |
| 重いレビュー必要 | 静的HTMLなので影響限定 |
| テスト範囲が広い | レイヤー分離でテスト範囲縮小 |

**結論:** 構造的にレビュー・テスト負担は軽減される

### 8-2. 工数・期間の見積もり

**懸念:**
> 70人日、半年〜1年かかるのでは

**対応:**
| 要因 | 説明 |
|------|------|
| 既存資産 | セクションビルダー、API群が実装済み |
| AI活用 | Claude Codeで作業加速 |
| 静的HTML | 検証負担が少ない |
| cursor.com事例 | 3日で移行完了の実績 |

**結論:** 2-4週間で主要移行完了可能

### 8-3. Takaさん一人で大丈夫か

**懸念:**
> リソースが足りないのでは

**対応:**
| 領域 | 担当 | 備考 |
|------|------|------|
| フロントエンド/API | Taka | 対応可能 |
| AI連携 | Taka | 対応可能 |
| DB設計 | Eng支援 | 1-2日 |
| セキュリティレビュー | Eng支援 | 必要に応じて |

**結論:** 大部分はTakaさんで対応可能。DB周りのみ支援必要

### 8-4. SEOが落ちないか

**懸念:**
> 移行でSEOを毀損するリスク

**対応:**
- URL構造完全維持
- 301リダイレクト設定
- GSC日次監視
- 段階的移行（並行運用期間あり）

**結論:** SEO維持は最優先事項として対応
