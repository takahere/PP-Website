WordPressから静的HTML + AI運用への移行により：
- **運用コスト削減**（年間¥100万以上）

## 2. 現状の課題

### 2-1. WordPressの根本的課題

| 課題 | 現状 | 影響 |
|------|------|------|
| **環境分散** | WP 2環境 + 別CMS（リリースノート） | 管理コスト増大 |
| **属人化** | 担当者ごとに運用方法が異なる | ナレッジ分断 |
| **プラグイン依存** | 変更の影響範囲が読めない | 改修リスク |
| **デザイン制約** | テーマ依存で自由度低い | ブランド表現に限界 |
| **セキュリティ** | PHP + プラグインの脆弱性リスク | 継続的な対応必要 |

### 2-2. AI時代の変化

```
【従来のコンテンツ運用】
人間がCMS GUI操作 → 手動で公開 → 手動で分析 → 手動で改善
                    ↓
            全てに人手が必要

【AI時代のコンテンツ運用】
人間が自然言語で指示 → AIが実行 → AIが分析 → AIが改善提案
                    ↓
            人間は承認のみ
```

**市場の変化:**
- HubSpotが「Loop Marketing」を提唱（2025年）
- Google検索の60%がクリックなしで終了
- 従来のファネル型マーケティングが機能不全

**機会:**
- AI-Firstで運用を再定義できる
- 先行者として市場ポジションを確立できる
- 自社課題解決がPRM製品の差別化になる

### 本提案の前提：コンテンツDBとしてのGitHub

本提案の根幹となる設計思想：

> **公開コンテンツのデータベースはGitHubリポジトリ内のMarkdownファイル。**
> **従来型のDBは、AI Content Operationsのための分析・インテリジェンス機能にのみ使用。**

```
┌─────────────────────────────────────────────────────────────┐
│  公開サイト（静的HTML）                                        │
│  ・記事、LP、ページ                                           │
│  ・ビルド時にMarkdownから生成                                  │
└─────────────────────────────────────────────────────────────┘
          ↑ ビルド時に読み込み
┌─────────────────────────────────────────────────────────────┐
│  コンテンツDB = GitHub Markdown                               │
│  /content/articles/*.md                                      │
│  /content/pages/*.md                                         │
│  ・バージョン管理はGit標準機能                                  │
│  ・レビューはPR、ロールバックはgit revert                      │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  Intelligence DB（Phase 4で追加、任意のプラットフォーム）        │
│  ・分析データ蓄積（GA4/GSCの時系列データ）                      │
│  ・AI学習データ                                               │
│  ・KPIダッシュボード用クエリ                                   │
│  ※公開サイトとは完全に分離された社内システム                    │
└─────────────────────────────────────────────────────────────┘
```

**なぜこの設計か：**
- 公開サイトは静的HTML。DBへのランタイム接続が不要
- コンテンツ変更 = Markdownファイル編集 → git commit → 自動ビルド
- Git = バージョン管理、レビュー、ロールバックすべて標準機能で完結
- Intelligence用DBは後から追加可能（公開サイトに影響しない）

## 3. 移行方針

### 3-1. 設計原則

> **「DBの役割は公開サイトの表示ではなく、インテリジェンス機能のデータ基盤」**
> **「公開コンテンツはGitHub Markdown、DBはインテリジェンス機能専用」**

この原則により：
- 公開サイトは静的HTMLで高速・安全
- コンテンツ管理はGit標準機能で完結
- 従来型DBはAI/分析のためにのみ使用（必要になってから追加）
- レビュー・テスト負担を構造的に軽減

### 3-3. なぜGitHub Markdownか

| 観点 | 従来型DB | GitHub Markdown |
|------|---------|-----------------|
| バージョン管理 | 自前実装必要 | Git標準機能 |
| 変更履歴 | 追加開発 | `git log` |
| レビューフロー | 自前実装 | Pull Request |
| ロールバック | 複雑 | `git revert` |
| コスト | DB費用発生 | ¥0 |
| AI/Claudeとの相性 | API経由 | ファイル直接編集（高速） |

### 3-4. Markdownでリッチなページは作れるか

**作れます。** MarkdownはデータのみでビジュアルはReactコンポーネントが担当。

```markdown
---
title: "PartnerProp"
template: "lp"
sections:
  - type: hero
    headline: "パートナービジネスを加速"
    background: { video: "/videos/hero.mp4" }
    animation: "fade-up"
  - type: features
    columns: 3
    items: [...]
---
```

→ Reactコンポーネントがこれを読み込み、Apple的なリッチLPをレンダリング

### 3-5. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| フロントエンド | Next.js 16 (App Router) | 静的生成 + 必要時のみ動的 |
| スタイリング | Tailwind CSS | 一貫性、パフォーマンス |
| コンテンツ | GitHub Markdown | Git標準機能でバージョン管理 |
| ホスティング | Vercel | Next.js最適化、グローバルCDN |
| AI | Claude API / OpenAI | コンテンツ生成、分析 |
| 監視 | Sentry | エラー追跡、パフォーマンス監視 |
| Intelligence DB | Supabase等（Phase 4〜） | 分析データ蓄積用、任意 |

### 3-6. 代替案との比較

| 選択肢 | メリット | デメリット | 判定 |
|--------|---------|-----------|------|
| **STUDIO移行** | 学習コスト低 | AI統合不可、また移行したくなる | ✗ |
| **HubSpot移行** | Loop Marketing対応 | HubL制約、$1,500+/月 | ✗ |
| **WP継続** | 変更なし | 課題解決せず、コスト増大 | ✗ |
| **静的HTML + AI** | AI統合、低コスト、高速 | 初期構築必要 | ✓ |

**なぜ自社開発か:**
- 既製品は「CMS + AI機能追加」（後付け）
- 提案は「AI-First + 静的HTML」（根本から異なる）
- cursor.com事例：3日で移行、運用コスト激減

## 4. SEO維持戦略

### 4-1. 基本方針

**SEO維持は最優先事項。**

| 項目 | 対応 |
|------|------|
| URL構造 | 完全維持（変更なし） |
| リダイレクト | やむを得ない変更は301で対応 |
| canonical | 全ページで正規URL設定 |
| sitemap | 移行後も正規URLのみ含める |

### 4-2. 移行時の対応

```
Phase 1: 並行運用
┌─────────────┐     ┌─────────────┐
│ WordPress   │     │ Next.js     │
│ (既存)      │     │ (新規)      │
└──────┬──────┘     └──────┬──────┘
       │                   │
       └─────────┬─────────┘
                 ↓
         Vercel Edge
         (パス別ルーティング)

・/lab/* → Next.js
・その他 → WordPress（当面）
```

```
Phase 2: 完全移行
┌─────────────────────────────────────┐
│            Next.js                  │
│  (全ページ静的HTML)                  │
└─────────────────────────────────────┘
         ↓
    Vercel CDN配信

・旧URLは全て301リダイレクト設定
・GSCで404/リダイレクトエラー監視
```

### 4-3. 監視体制

| 指標 | ツール | 頻度 |

## 8. 想定される懸念と対応策

### 8-1. 品質評価コストの増大

**懸念:**
> フルスクラッチで作ると品質評価コストが増大する

**対応:**
| 従来のCMS | 提案アーキテクチャ |
|-----------|------------------|
| DB変更→即公開 | DB変更→ビルド実行→公開 |
| 重いレビュー必要 | 静的HTMLなので影響限定 |
| テスト範囲が広い | レイヤー分離でテスト範囲縮小 |

**結論:** 構造的にレビュー・テスト負担は軽減される

### 8-2. 工数・期間の見積もり

**懸念:**
> 70人日、半年〜1年かかるのでは

**対応:**
| 要因 | 説明 |
|------|------|
| 既存資産 | セクションビルダー、API群が実装済み |
| AI活用 | Claude Codeで作業加速 |
| 静的HTML | 検証負担が少ない |
| cursor.com事例 | 3日で移行完了の実績 |

**結論:** 2-4週間で主要移行完了可能

### 8-3. Takaさん一人で大丈夫か

**懸念:**
> リソースが足りないのでは

**対応:**
| 領域 | 担当 | 備考 |
|------|------|------|
| フロントエンド/API | Taka | 対応可能 |
| AI連携 | Taka | 対応可能 |
| コンテンツ設計 | Taka | 不要（Markdown構造のみ） |
| Intelligence DB設計 | Taka + Eng | Phase 4で1-2日の支援 |
| セキュリティレビュー | Eng支援 | 必要に応じて |

**結論:** Phase 1-3は完全にTakaさんで対応可能。Phase 4のDB設計時のみ支援を依頼。

### 8-4. SEOが落ちないか

**懸念:**
> 移行でSEOを毀損するリスク

**対応:**
- URL構造完全維持
- 301リダイレクト設定
- GSC日次監視
- 段階的移行（並行運用期間あり）

**結論:** SEO維持は最優先事項として対応

### 8-5. 障害時に対応できるのか

| 想定障害 | WordPress | 本提案 |
|---------|-----------|-------------|
| サーバーダウン | 自前復旧 or 外注 | Vercel自動復旧（SLA 99.99%） |
| コンテンツ障害 | DBバックアップから復旧 | `git revert`で即時ロールバック |
| Intelligence DB障害 | - | 公開サイトに影響なし（分離されている） |
| 表示崩れ | プラグイン調査 | Git revertで即時ロールバック |

**ポイント:**
- 公開サイトはGitHub + Vercelのみ。DBに依存しないため障害に強い
- Intelligence DBに障害があっても公開サイトは影響を受けない
